1. Executive Summary
This document is the complete technical specification for the Makerere University Technology and Innovation Center (Mak-TIC) Innovation Portal. It defines every workflow, database schema, API endpoint, page, middleware, integration, and AI automation layer required to build a production-ready system.
The system will be built on Next.js 16 (App Router) deployed to Vercel, with Supabase as the database, authentication, storage, and real-time engine. The frontend uses React 19 with Tailwind CSS v4 and shadcn/ui components. Agentic AI workflows are powered by Google Gemini API (via Google AI Studio) calls through Vercel AI SDK.
1.1 Core Objectives
Allow innovators (Makerere students, staff, alumni, and external applicants) to register, submit projects, upload documents, and track progress through a self-service portal.
Provide a structured stage-gate review pipeline for center staff to evaluate, score, and advance projects.
Automate triage, scoring, mentor matching, SDG tagging, and duplicate detection using agentic AI.
Manage funding disbursement, facility booking, IP tracking, and funder reporting.
Enable open innovation challenges, community forums, and a public innovation showcase.
Integrate with external systems: mobile money payments, UNDP reporting feeds, AfriLabs network, and Makerere AIMS.

2. System Architecture Overview
The system follows a 4-tier architecture: Applicant Portal (client), API Layer (Next.js Route Handlers + Supabase Edge Functions), Intelligence Layer (AI agents), and Data Layer (Supabase PostgreSQL + Storage). All tiers communicate through a typed API contract defined with Zod schemas that are shared between client and server.

2.2 Key Architectural Decisions
Server Components by default: all data-fetching pages are React Server Components. Client components only where interactivity is needed (forms, modals, real-time).
Row-Level Security (RLS) on every Supabase table: the database enforces access control, not just the API layer. Even if middleware is bypassed, data remains protected.
Edge Middleware for auth: proxy.ts runs on Vercel Edge and validates the Supabase session cookie on every request. Unauthenticated users are redirected to /login.
AI agents run as background jobs: long-running AI tasks (scoring, scanning) are triggered via Supabase Edge Functions with pg_cron or Supabase Queues, not blocking the user request.
Optimistic UI with real-time sync: when an innovator submits a form, the UI updates immediately. Supabase Realtime pushes the confirmed state back to all connected clients.
File uploads use Supabase Storage with signed URLs: files are uploaded directly from the client to Supabase Storage (bypassing the server), then the signed URL is stored in the database.

3. Database Schema (Supabase PostgreSQL)
All tables use UUID primary keys generated by gen_random_uuid(). Timestamps use timestamptz. Every table has Row-Level Security (RLS) enabled. Soft deletes are used where data retention is required (deleted_at timestamptz NULL). The schema is organized into logical domains: Users, Projects, Reviews, Challenges, Community, Resources, Funding, Facilities, and System.

4. Middleware, API Routes & Edge Functions
4.1 Next.js Proxy (proxy.ts)
Runs on Vercel Edge for every request. Performs three functions:
Session validation: reads the Supabase auth cookie (sb-access-token), verifies the JWT, and attaches the user object to the request. If invalid or expired, redirects to /login.
Role-based route protection: checks the user role against the requested path. /admin/* requires role in {admin, reviewer, lab_manager}. /portal/* requires any authenticated user. /showcase/* and /investors/* are public.
Rate limiting: uses Vercel KV (Redis) to enforce rate limits per IP and per user. API routes are limited to 100 requests/minute for authenticated users, 20/minute for unauthenticated.
4.2 API Route Handlers
All API routes live under /app/api/ and follow REST conventions. Every route validates input with Zod, checks permissions via Supabase RLS, and returns typed JSON responses. Error responses follow a standard format: { error: string, code: string, details?: object }.

4.3 Supabase Edge Functions (Background Jobs)
Long-running or scheduled tasks run as Supabase Edge Functions, triggered by database webhooks, pg_cron, or Supabase Queues. These keep the user-facing API fast.
Function
Trigger
Purpose
ai-score-project
Database webhook on projects INSERT
Calls Gemini API to generate AI viability score, summary, and SDG tags for new submissions. Writes results back to projects table.
ai-match-mentor
Cron: daily at 06:00 EAT
Scans unmatched projects and available mentors. Uses Gemini to evaluate compatibility based on sector, stage, skills, language, and availability. Creates mentor_assignments.
ai-duplicate-check
Database webhook on projects INSERT
Computes embeddings of new project description using pgvector and finds cosine-similar projects above 0.85 threshold. Flags matches in project_comments.
review-deadline-reminder
Cron: daily at 08:00 EAT
Checks review_assignments for approaching deadlines (48h) and overdue reviews. Sends email + in-app notification.
milestone-deadline-check
Cron: daily at 07:00 EAT
Checks project_milestones for overdue milestones. Updates status to overdue. Notifies project owner and admin.
funding-auto-release
Database webhook on project_milestones UPDATE (completed_at)
When a milestone is verified, checks if a funding_tranche is linked. If so, initiates the mobile money disbursement.
momo-disbursement
Called by funding-auto-release
Calls MTN MoMo API to send funds to the innovator's phone number. Updates funding_tranches status.
generate-funder-report
Cron: 1st of each month
Aggregates data per funding_source and generates PDF or structured JSON report. Emails to funder contact.
sync-showcase
Cron: hourly
Checks projects with is_public = true and generates/updates cached showcase data for the public portal.

5. Agentic AI Specifications (Google Gemini)
This section defines every AI agent in the system: its purpose, inputs, outputs, Gemini API configuration, prompt structure, and how it integrates into the workflow. All agents use the Google Gemini API via the Vercel AI SDK (@ai-sdk/google provider). Primary model: gemini-2.5-flash (free tier: 1500 RPD, 1M TPM). Fallback for complex reasoning: gemini-2.5-pro (free tier: 25 RPD). Production paid tier: Google AI Studio at $0.15/1M input tokens for Flash.
5.0 Why Gemini + Google AI Studio
Google AI Studio provides the most generous free tier in the industry for a university project. For prototyping and early production with moderate traffic, the entire AI layer can run at zero cost. This section documents the model selection strategy, free tier limits, and upgrade path.

5.0.3 Integration Architecture (Code Pattern)
All AI calls use the Vercel AI SDK with the @ai-sdk/google provider. This gives a provider-agnostic interface: if Gemini's free tier changes or another model becomes better value, swapping is a one-line code change.
// Install: npm install ai @ai-sdk/google
import { google } from '@ai-sdk/google';
import { generateObject, generateText, streamText } from 'ai';

// Structured output (scoring, tagging, matching):
const result = await generateObject({
  model: google('gemini-2.5-flash'),
  schema: projectScoreSchema, // Zod schema
  prompt: `Evaluate this project: ${projectData}`,
});

// Streaming chat (quality assistant):
const result = streamText({
  model: google('gemini-2.5-flash'),
  messages: conversationHistory,
});

// Google Search grounding (IP scan):
const result = await generateText({
  model: google('gemini-2.5-pro', {
    useSearchGrounding: true,
  }),
  prompt: `Search for prior art: ${desc}`,
});

// Embeddings (duplicate detection):
import { embed } from 'ai';
const { embedding } = await embed({
  model: google.textEmbeddingModel('text-embedding-004'),
  value: projectDescription,
});
Environment variable: GOOGLE_GENERATIVE_AI_API_KEY (free from aistudio.google.com, no credit card needed).
5.0.4 Gemini-Specific Advantages
Google Search grounding is built into the Gemini API at no extra cost. Critical for the IP Landscape Scanner agent. With other providers, you would need a separate search API subscription.
Structured output via responseSchema guarantees valid JSON. No retry logic for malformed responses. Every scoring and matching agent returns a Zod-validated object directly.
Multilingual: Gemini handles English, Luganda, and Swahili content well, important for Makerere where descriptions may include local language terms.
Long context (1M tokens Flash, 2M Pro): entire project portfolios can be analyzed in a single call for cross-project insights.
Batch API: for monthly reports and bulk re-scoring, Gemini batch API offers 50% discount from already-low rates.
Provider-agnostic code: if the team later wants to use Claude or GPT for specific tasks, the Vercel AI SDK swap is trivial.

5.1 Agent: Project Scorer
Purpose
Automatically evaluates new project submissions and generates a viability score, structured summary, and SDG tags. This runs within 30 seconds of submission, so reviewers see AI pre-analysis before they begin their own evaluation.
Trigger
Database webhook fires when a new row is inserted into the projects table. The Edge Function ai-score-project reads the full project data (title, description, problem_statement, proposed_solution, sector, track, support_needed) and calls Gemini.
Gemini API Call Configuration
Model: gemini-2.5-flash (free tier) or gemini-2.5-pro (complex tasks)
Max tokens: 2000
Temperature: 0.2 (low creativity, high consistency). responseSchema for guaranteed JSON output.
System prompt instructs Gemini to act as an expert innovation evaluator at an East African university innovation center.
Prompt Template
You are an expert innovation evaluator at Makerere University's
Technology and Innovation Center in Kampala, Uganda. Evaluate the
following project submission and return a JSON object with:
1. score (0-100): viability score based on problem clarity, solution
   feasibility, team potential, innovation level, and market fit
2. summary (2-3 sentences): executive summary for reviewers
3. sdg_tags (array of integers 1-17): relevant SDGs
4. sdg_reasoning (1 sentence per SDG): why each SDG applies
5. strengths (array of strings): top 3 strengths
6. concerns (array of strings): top 3 concerns or gaps
7. recommended_support (array): which support types are most critical
Return ONLY valid JSON, no markdown or preamble.
Output Handling
The JSON response is parsed and written to the projects table: ai_score, ai_summary, sdg_tags, ai_sdg_reasoning. A notification is sent to the project owner confirming submission and showing the AI-suggested SDG tags (score is internal only, not shown to applicants).
5.2 Agent: Mentor Matcher
Purpose
Matches unmatched projects with the best available mentor based on sector alignment, stage, language, workload, and historical outcomes. Runs daily and can be triggered manually by admins.
Input
All projects with stage in (problem_validation, solution_viability, impact_assessment) that have no active mentor_assignment, plus all mentor_profiles where current_mentees < max_mentees.
Gemini API Prompt
Given the following unmatched projects and available mentors,
propose optimal pairings. For each pairing, provide:
1. project_id and mentor_id
2. compatibility_score (0-100)
3. reasoning (1 sentence explaining the match)
Prioritize: sector expertise match > stage experience > language >
workload balance. Return JSON array sorted by compatibility_score.
Output
Creates mentor_assignment records (status: proposed). Sends notification to both mentor and project lead. Admin can approve or override before the assignment becomes active.
5.3 Agent: Duplicate Detector
Purpose
Identifies projects that substantially overlap with existing or past projects. Uses a two-stage approach: first a fast vector similarity search (pgvector), then a Gemini confirmation for high-similarity matches.
Stage 1: Vector Search
On project INSERT, the Edge Function generates an embedding of the project description using a sentence-transformer model (via Supabase pgvector extension). It queries the project_embeddings table for cosine similarity > 0.82. If matches are found, proceed to Stage 2.
Stage 2: Gemini Confirmation
Compare these two project descriptions. Are they working on
substantially the same problem with a similar approach?
Return JSON: { is_duplicate: boolean, similarity: 0-100,
  explanation: string, collaboration_potential: string }
Output
If is_duplicate is true, a project_comment is auto-created flagging the overlap. The admin pipeline shows a duplicate alert badge. If collaboration_potential is high, the system suggests the two teams connect.
5.4 Agent: IP Landscape Scanner
Purpose
Searches patent databases and academic literature to identify prior art relevant to a project. Triggered manually by admin or by the IP office when evaluating an invention disclosure.
Implementation
Uses Gemini with Google Search grounding enabled (free built-in tool) to search Google Patents, WIPO PATENTSCOPE, and Google Scholar for related technologies. Returns a structured report with: number of related patents found, key differentiators, freedom-to-operate assessment (basic), and recommended next steps.
5.5 Agent: Application Quality Assistant
Purpose
A real-time assistant that helps innovators improve their application before submission. Operates as a chat interface on the project submission page. The innovator can ask questions like 'Is my problem statement clear enough?' or 'What am I missing?' and get instant feedback.
Implementation
Uses the Vercel AI SDK's useChat hook with streaming (via @ai-sdk/google provider). The system prompt constrains Gemini to only provide feedback on the application content and not generate the content itself. It should coach, not write.
5.6 Agent: Report Generator
Purpose
Generates narrative funder reports from raw data. Takes aggregate statistics (project counts, stage distribution, SDG breakdown, gender ratios, expenditure) and produces a polished narrative report suitable for UNDP or other donors.
Implementation
Admin triggers report generation for a specific funder and date range. The system aggregates data from the database, passes it to Gemini with the funder's reporting template (stored in funding_sources.reporting_template), and Gemini generates a structured report. The report is then rendered as a PDF via React-PDF and emailed to the funder.

7. External Integrations
7.1 Mobile Money (MTN MoMo + Airtel Money)
Integration approach
Use the MTN MoMo API (Collections for receiving, Disbursements for sending) and Airtel Money API for funding disbursements. Both APIs require partner registration and API key provisioning in Uganda.
Disbursement workflow
Admin verifies a milestone is complete and approves the funding tranche.
The system calls the MoMo Disbursement API with the innovator's phone number and amount.
MoMo processes the payment and sends a callback to /api/webhooks/momo.
The webhook handler updates funding_tranches.status to disbursed and logs the transaction reference.
If the payment fails, the system retries up to 3 times with exponential backoff, then alerts the admin.
7.2 Email (Resend + React Email)
Transactional email for: registration confirmation, password reset, review assignment notifications, stage advancement notifications, funding disbursement confirmation, milestone deadline reminders, mentor session reminders, challenge launch announcements. Templates built with React Email for consistent branding. Sent via Resend API (Vercel-integrated).
7.3 Makerere AIMS (Academic Information Management System)
Future integration to auto-verify student enrollment and pull researcher profiles. For the initial build, affiliation is self-declared with manual verification by admin. The system is designed with an aims_verified boolean on profiles that defaults to false and can be set to true by admin or by a future AIMS API integration.
7.4 UNDP / Timbuktoo Reporting Feed
The system generates structured reports in the format required by UNDP's Timbuktoo initiative. Reports are generated as JSON data (matching their template) and as PDF documents. For now, reports are manually downloaded and submitted. Future: direct API integration with UNDP's reporting system when their API becomes available.
7.5 AfriLabs Network
Projects with is_public = true are synced to a public API endpoint (/api/public/showcase) that can be consumed by AfriLabs' innovation directory. The API returns: project title, description, sector, SDG tags, team, stage, and contact information. Format: JSON API with pagination.
7.6 Supabase Realtime
Used for: live notification delivery (notification bell updates without refresh), pipeline board updates (when a project advances, all admin clients see the card move in real-time), challenge vote counts (live updating), booking calendar updates. Implemented via Supabase Realtime subscriptions on relevant tables.

8.2 File Upload Security
File uploads go directly to Supabase Storage using signed upload URLs (generated server-side).
Allowed MIME types: application/pdf, application/vnd.ms-powerpoint, application/vnd.openxmlformats-officedocument.presentationml.presentation, video/mp4, video/webm, image/jpeg, image/png, application/vnd.openxmlformats-officedocument.spreadsheetml.sheet.
Maximum file size: 50MB per file, 200MB total per project.
Storage bucket policies: project files are private by default. Signed read URLs with 1-hour expiry are generated on demand.
Public showcase images are in a separate public bucket with CDN caching.
8.3 Data Governance
All innovators retain full IP rights to their submissions. The center's terms of use explicitly state this.
Project data is never shared with third parties without the innovator's explicit consent (is_public flag).
Audit log captures all data access and modifications with actor, timestamp, and IP address.
Soft deletes ensure data is never permanently lost. Hard deletes require admin action and are logged.
Database backups run daily (Supabase managed) with 30-day retention.
All API calls to the Gemini AI include a disclaimer that project data is sent to Google's API for processing. Users consent to this during registration.

11. Development Phases & Sprint Plan
The project is structured in 4 phases over approximately 12-16 weeks. Each phase produces a demoable increment.
Phase 1: Foundation (Weeks 1-3)
Goal: Working registration, login, project submission, and basic admin pipeline.
Set up Next.js 16 project with TypeScript, Tailwind v4, shadcn/ui.
Configure Supabase project: database schema (Users + Projects domains), storage buckets, RLS policies.
Build auth flow: /login, /register (multi-step), /forgot-password.
Build proxy.ts for auth + route protection.
Build /dashboard (innovator home).
Build /projects/new (multi-step wizard with file upload).
Build /projects/[id] (project detail with tabs).
Build /admin/pipeline (Kanban board with drag-and-drop stage advancement).
Integrate AI Project Scorer agent (fires on project submission).
Demo deliverable: An innovator can register, submit a project with uploads, see AI scoring results, and an admin can view and advance the project through pipeline stages.
Phase 2: Intelligence & Review (Weeks 4-7)
Goal: Complete review pipeline, mentor system, and all AI agents.
Build stage_gate_reviews system: reviewer assignment (auto + manual), scoring rubric, multi-reviewer aggregation.
Build /admin/scoring (AI vs human score comparison).
Build mentor system: mentor_profiles, mentor_assignments, session tracking.
Integrate AI Mentor Matcher agent.
Integrate AI Duplicate Detector (pgvector setup + Gemini confirmation).
Build Application Quality Assistant (chat interface on submission page).
Build notification system (in-app + email via Resend).
Build audit_log infrastructure.
Demo deliverable: Full review pipeline with AI scoring, mentor matching, duplicate detection, and email notifications working end-to-end.
Phase 3: Ecosystem (Weeks 8-11)
Goal: Challenges, community, LMS, facility booking, and funding management.
Build /challenges system (create, submit, vote, judge).
Build /community forum (categories, posts, threaded replies).
Build /resources LMS (courses, modules, progress tracking, certificates).

Build /admin/funding (sources, allocations, tranches, disbursement).

Build /admin/ip (IP tracking dashboard).
Integrate AI IP Landscape Scanner.
Demo deliverable: Complete ecosystem with challenges, forum, courses, facility booking, and mobile money disbursements.
Phase 4: Public-Facing & Polish (Weeks 12-14)
Goal: Public showcase, investor portal, analytics, and production hardening.
Build / (landing page).
Build /showcase (public innovation portfolio).
Build /investors (investor-facing portal with gated access).
Build /admin/reports (comprehensive analytics dashboard with charts).

12.2 Supabase Database Trigger Examples
Auto-create profile on signup
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, email, full_name, role, affiliation)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'full_name', ''),
    'innovator',
    COALESCE(NEW.raw_user_meta_data->>'affiliation', 'external')
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
Auto-generate project slug
CREATE OR REPLACE FUNCTION public.generate_project_slug()
RETURNS trigger AS $$
BEGIN
  NEW.slug := lower(regexp_replace(NEW.title, '[^a-zA-Z0-9]+', '-', 'g'))
    || '-' || substr(NEW.id::text, 1, 8);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_project_slug
  BEFORE INSERT ON projects
  FOR EACH ROW EXECUTE FUNCTION public.generate_project_slug();
